Politica de preço (JSON)

O Distribuidor que optar pelo desenvolvimento deste item ao invés do preço XML, poderá trabalhar com preços diferenciados por cliente, devido a possibilidade de enviar N políticas de preços, cada politica possui os produtos, tabelas de preços e grupo de clientes que deseja vincular. Uma vez que a plataforma permite trabalhar com várias políticas, é possível aplicar os preços diferenciados.

As Politicas de preços trabalham no protocolo REST, onde a plataforma disponibiliza ao seller duas APIS sendo elas:

- API GET para que o seller resgataro Token de autenticação;

- API POST para realizar o envio da politica de preços, utilizando o token como autenticação.

Detalhes sobre regras de negócios para políticas de preços


A politica com ID "DEFAULT" que veremos nos exemplos abaixo, se aplicará para todos os clientes que não fazem parte de outra politica de preços que o seller irá enviar. Ou seja, quando um novo cliente realizar o cadastro na plataforma, automaticamente estará na politica Default, e praticará os preços que foram enviados para ela (os preços a prazo, só estarão disponíveis para clientes que possuirem status de aprovação e limite de crédito concedido). Tendo em vista esse cenário, não há a necessidade de incluir todos os CNPJs que farão parte da politica DEFAULT. A plataforma considera que todos os CNPJs restantes que não estão relacionados a nenhuma outra politica, ingressem na DEFAULT.

O distribuidor pode enviar outras políticas além da politica default, atribuindo para cada política: produtos, tabelas de preços e clientes, para que assim, pratique preços diferenciados para os seus clientes conforme a necessidade do negócio.

Regras: Um cliente só pode fazer parte de uma política de preços, sendo assim: caso o mesmo esteja em uma política diferente do default, o mesmo enxergará apenas os preços que foram enviados para ela.

- As requisições para plataforma, não devem ultrapassar 50 produtos por requisição, para que o pacote de integração não onere a fila da plataforma, o mesmo vale para inclusão de inclusão de novos clientes para a mesma.

- As políticas de preços trabalham em forma de atualização, ou seja: Caso o distribuidor envie a primeira requisição para política default contendo 50 produtos, e uma outra requisição contendo somente 1 produto, a política irá conter 51 produtos. Isso vale para inclusão de clientes na política: Caso houver a necessidade de incluir um ou um grupo de clientes em uma determinada política já existente, não há a necessidade de enviar as requisições de clientes e produtos, deve ser enviado somente os clientes para a política, com isso ganhamos performance.

Apoio para o desenvolvimento

O Web Service responsável pela integração de estoque é o StockServices.

A operação responsável pela integração de estoque é a setStock.

Esse WebService fica hospedado no Middleware, portanto, deve ser chamada pelo Distribuidor. O material da apoio para construção dos WSDL e XSD estão presentes no item 11 do manual.
